浏览器缓存

发表于2023-06-03|更新于2023-06-04
|阅读量:

why

前端开发注重性能和用户体验，对于性能最简单的评判就是网站打开到可交互的渲染速度。缓存策略做好可以使用下载好资源重复利用，减少请求次数，减少带宽，降低网络负荷。

what

web缓存主要针对一些web资源(html\css\js\数据)，介于服务端和浏览器之间的数据副本。首次打开网页，浏览器向服务端请求必要资源，如果使用了web缓存，下一次访问网页根据缓存策略，决定是否直接使用缓存中的资源，还是再次请求新的资源。以此减少客户端、服务端通信延迟。

作用

  1. 减少网络带宽

  2. 减少服务器压力

  3. 减少网络延迟，加快页面打开速度

缺点

  1. 占用硬盘空间

  2. 页面缓存可能会导致页面布局、图片、脚本未及时更新

  3. 缓存没清理机制

浏览器缓存种类

  1. http缓存是基于HTTP协议的浏览器文件级缓存机制
  2. websql只有较新chrome支持，独立规范出现
  3. indexDB为了客户端储存客观数量的结构性数据(异步获取，容量较大，用户不清除)
  4. local、session
  5. application cache大部分静态资源
  6. cacheStorage在serviceWorker规范定义，保存ServiceWorker声明的cache对象
  7. flash缓存

无法被缓存

  1. http头部cache-control:no-cache，pragma:nocache、cache-control:max-age=0
  2. 根据cookie，认证信息决定输入内容是够可以被缓存
  3. https加密请求
  4. post请求
  5. 响应头不存在last-modified/Etag和cache-control/expires等

缓存位置

从位置来说，各自有优先级，一次查找没有缓存时去请求网络

1. service Worker

协议必须是HTTPS(涉及到请求拦截所以要保障安全).运行在浏览器背后的独立线程，一般实现缓存功能。与浏览器的内建缓存不同，可以自由控制缓存哪些文件、如何匹配缓存、如何读取缓存，并且持续性缓存。
一般三个步骤可以实现缓存: 首先注册Service Worker，然后监听到install事件以后缓存需要缓存的文件。下次用户访问则通过拦截请求查询是否存在缓存。存在则直接读取缓存文件，否则去请求。
当Service Worker没有命中缓存，则需要fetch、ajax函数调用资源。也就是说如果没有在Service Woker内命中缓存，无论我们是在Memory Cache还是在网络请求中获取数据，浏览器显示都是在Service Woker中获取。

2. Memory Cache

内存中的缓存，主要包含了本页面已经抓取的资源(图片、css、js等)。读取内存数据肯定比磁盘快，但是缓存持续性较短。随着进程的释放而释放。

可以看出访问一次页面后,刷新页面大部分资源都是来自于Memory Cache。
内存缓存很重要的缓存资源是preloader相关(<linkrel=”prefetch” >).相关指令是页面优化常见手段之一，可以一边解析js\css一边来请求网络资源
注意的是，内存缓存在缓存资源时候不关心返回http缓存头cache-Control是什么值，资源匹配也不仅仅是URL的匹配，还会对Content-Type、CORS其他特征校验。

3. disk Cache

硬盘缓存，读取速度较慢，但容量及时效性胜过于Memory Cache。
在所有浏览器缓存，Disk Cache覆盖面最大。会根据HTTP Header字段判断哪些资源需要缓存，哪些直接使用，那些过期需要重新请求。即使是跨站点，相同地址资源被缓存也不会再次去请求。绝对部分缓存来自于Disk Cache.

  1. 对于大文件来说，大概率是不会存储在Memory Cache
  2. 当前内存使用率较高，文件优先存储Disk

4. push Cache

推送缓存是HTTP/2的内容，当Service Worker\Memory Cache\Disk Cache都没有命中，会被使用。只在Session中存在，会话结束则被释放，缓存时间短暂。Chrome中约为5分钟左右。并非严格执行HTTP 头中指令。
国内资料很少，HTTP/2并不普及

  1. 所有资源都能被推送和缓存，但是Edge\safari支持较差
  2. 可以推送no-cache和no-store资源
  3. 一旦连接关闭则被释放
  4. 多个页面可以使用同一个HTTP/2连接，也意味着可以同一个Push Cache，依赖浏览器来实现，有的浏览器会对相同域名但不同tab标签使用同一个HTTP连接。
  5. Push Cache只能被使用一次
  6. 浏览器可以拒绝接受已经存在的资源推送
  7. 可以给其他域名推送资源

如果以上四种缓存都没有命中，则使用网络请求获取资源。
为了性能，大部分接口选择好缓存策略。通常浏览器缓存策略分为强缓存和协商缓存，策略是通过HTTP Header来实现。

缓存过程分析

浏览器与服务端通信为应答模式。浏览器发起HTTP-服务端相应请求。

上图可得知：

  1. 浏览器每次发送请求，现在浏览器缓存中查找请求结果和缓存标识
  2. 浏览器每次拿到返回结果都会将返回结果和缓存标识存入浏览器缓存中。

强缓存

不会向服务器发送请求，直接从缓存中读取资源。Chrome的network可以看到请求返回200状态码，Size为From disk Cache / from Memory Cache。
强缓存的实现通过设置HTTP Header实现

1. Expires

缓存过期时间，指定资源到期时间。也就是说Expires=max-age + 请求时间。需要和last-modified结合使用。Expires是web服务端响应消息，相应http请求时告诉浏览器过期时间前可以从浏览器缓存直接读取数据。
Expires是HTTP/1产物，受限于本地时间，如果修改则会失效.
Expires：Wed.22Oct202210:41:00GMT表示资源会在此时间后过期

2.Cache-control

HTTP/1.1,Cache-control是最重要的规则，用户控制网页缓存。比如Cache-control=max-age=300,在请求正确返回时间(浏览器记录下来)的5分钟内再次加载资源是命中强缓存的。
cache-control在请求头或者响应头设置，可以组合使用指令

1. public

所有内容都将被缓存(客户端和服务端)，可被任何阶段缓存。

2. private(默认取值)

只可以被客户端缓存

3. no-cache

客户端缓存内容是否缓存则需要通过协商缓存校验决定。表示不使用Cache-control做前置校验，而是使用Etag或者Last-modified字段控制缓存。设置no-cache不是说不缓存数据，而是浏览器使用缓存数据前先确认下数据与服务端是否一致。

4. no-store

所有内容不会被缓存。不使用强缓存也不适用协商缓存。

5. max-age

缓存在 max-age=xxx 秒后失效

6. s-maxage

同max-age，只在代理服务生效。比如s-maxage=60,在60s内即使更新了CDN内容，浏览器也不会请求。 max-age普通缓存，s-maxage用于代理缓存，s-maxage优先级高。会覆盖掉max-age和Expires Header

7. max-stale

容忍的最大过期时间。 客户端愿意接受一个过期的缓存，如果指定了max-stale，则最大容忍时间为对应的秒数，如果没有指定，那么说明浏览器愿意接受任何age的响应。

8. min-fresh

能够容忍的最小新鲜度。 min-fresh标识了客户端不愿意接受新鲜度不多于当前age+min-fresh设定的时间之和的响应。

上图看到，可以多个指令配合

Expires和Cache-control对比

Expires是HTTP/1产物，Cache-control是HTTP1.1的产物。Cache-control优先级高于Expires。Expires其实是过时产物，只是兼容性写法。
强缓存判断依据是文件是否缓存过期标识。如果获知服务端内容发生变化则需要用到协商缓存策略。

协商缓存

协商缓存是强缓存失效之后，浏览器携带缓存标识向服务器发送请求，由标识来决定是否使用缓存的过程。
协商缓存有两种情况：

  1. 返回304和Not modified

  2. 返回200和请求结果

    通过设置HTTP Header实现： Last-modified 和 ETag

1. Last-modified 和 If-Modified-Since

浏览器第一访问资源在返回资源同时，在response Header中添加Last-modified的header，值是资源在服务器上最后的修改时间，浏览器接收后缓存资源和header。
Last-Modified: Fri, 22 Jul 2016 01:47:00 GMT
下次访问检测到有Last-modified，则添加If-modified-Since这个Header。值就是Last-modified。 再次收到请求，根据值与服务器最后的资源最对比，没有变化返回304和空响应体，从缓存中读取。如果值早于最后修改时间则返回新的资源和200.
弊端：

  • 本地打开缓存文件，即使没有修改也会修改Last-modified，服务端不能命中缓存并发送资源

  • 以秒计时，如果快速修改文件则资源命中缓存，不会返回新的资源。
    Last-modified是根据文件修改时间来判断，而ETag则是根据文件内容

2. ETag和If-None-Match

ETag是响应请求时返回当前资源的一个唯一标识。(服务端生成)，资源有变化时ETag会重新生成。浏览器请求会将上次请求的ETag放在If-None-Match里，比较客户端的ETag与If-None-Match是否一致，判断对于客户端是否被修改过。一致返回304不一致返回200及新的资源文件。

对比两种

  1. 精确度上ETag高

  2. 性能上只依靠时间的Last-modified更好，而ETag通过算法计算一个hash值

  3. 优先级上，ETag更高。

缓存机制

强缓存优先于协商缓存。强制缓存生效则直接使用缓存，不生效则协商缓存。由服务端决定是否使用缓存，协商缓存失效返回200和新的资源并存入缓存，生效则返回304直接使用缓存。
如果什么都没有设置，则浏览器通常会取响应头部的Date减去Last-modified值的10%作为缓存时间。

实际场景缓存策略

1. 频繁变动

2. 不常变化资源

如果想要强缓存失效(或者说是不使用缓存)则一般在文件路径或者页面路径添加hash值或者版本号等动态字符达到效果
(jquery.3.3.1.min.js)均采用这个模式。

用户行为对浏览器缓存的影响

  1. 打开网页，输入url，查找Disk Cache，没有匹配则请求网络

  2. 普通刷新，页面未关闭，首先查找Memory Cache 其次查找 Disk Cache

  3. 强制刷新，不使用缓存 Cache-Control:no-cache(兼容还带了Pragma:no-cache)服务直接返回200及文件资源。
